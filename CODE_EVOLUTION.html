  • Docker Compose deployment from day one

🔬 Observed Patterns & Assessment

Consistent strengths

  • Architecture clarity improved rapidly and was maintained — no regression to monolith patterns once the six-binary structure was established.
  • SPEC.md as a contract is genuinely enforced. CLAUDE.md files explicitly prohibit endpoints not in SPEC.md, and the prohibition is effective.
  • Test-to-source ratio exceeded 1.0 — test code now outweighs implementation code, a rare property in research projects.
  • Every documented failure in EXPERIENCES.md has a corresponding generalised lesson. The pre-flight checklist prevents known regressions.
  • Dependency minimalism is strict. No external Go dependencies outside stdlib. Dashboard uses minimal npm. CA crypto is stdlib-only.
  • Backward compatibility was maintained across all API changes. Experiments 1–7 were not broken by the AH5 expansion.

Areas warranting attention

  • G2/G3 (credential verification and token strength) remain open after 43 days. These are documented but not prioritised, which is appropriate for a research system — but worth a conscious decision before any external deployment.
  • No CI pipeline. Tests run manually via go test ./... and test-system.sh. A single go test invocation is fast but relies on developer discipline.
  • Bash test scripts in experiments use text-pattern matching (grep on HTTP responses), which is fragile. EXP-002 and EXP-004/006 document bugs caused by exactly this. Consider moving to Go-based integration tests in experiment-7 style.
  • The dual API surface in ServiceRegistry (legacy + AH5 on one port) is a clean short-term solution, but creates potential for consumer confusion. Long-term, migration guidance for new consumers to use AH5 endpoints would help.
  • No pagination on management query endpoints. Acceptable for in-memory use, but the AH5 spec mentions pagination. Noted in GAP_ANALYSIS but not scheduled.
  • node_modules were committed to VCS in several commits, inflating commit diffs (e.g., 6 186 files changed in one commit). This is inconvenient for history analysis.

Architecture evolution pattern

  • The repository exhibits a "converge and govern" pattern: rapid exploration, then explicit governance rules codified in CLAUDE.md to prevent drift.
  • Experiments are used as disposable sandboxes that don't pollute core. The rule "no import of core/internal from experiments" is strictly enforced.
  • The GAP_ANALYSIS is a living document that distinguishes between gaps (things not yet done), decisions (deliberate design choices), interpretations (ambiguous spec, chosen reading), and ambiguities (still unclear). This taxonomy is unusually mature.
  • Security was added incrementally without breaking changes — tlsutil is opt-in via environment variables, not required. This is a good pattern for research infrastructure.

What the commit history reveals

  • The "Claude optimization" commits (May 5, May 8) introduced governance files and workspace structure — not feature code. This shows maturation: the repository needs managing, not just building.
  • Stage commits within a day (experiment-7 had 9 stages in one day) indicate tight TDD loops: test → implement → verify → commit.
  • EXPERIENCES.md was created after the repo was well-established — the team recognised the value of a failure log from accumulated experience, not as an afterthought added on day one.
  • The separation of concerns is enforced not just architecturally but temporally: core was stable before experiments began, experiments were stable before mTLS was added.
